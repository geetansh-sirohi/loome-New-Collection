# 🧵 LOOME — Luxury Fashion Landing Page
 
A luxury-inspired fashion landing page built using **only HTML & CSS — no JavaScript.**
Created as part of **Sheryians Coding School, Cohort 3.0** assignment.
 
---
 
## 🚀 Project Overview
 
LOOME was inspired by premium fashion houses like **Gucci** and **Louis Vuitton** — brands that treat their websites as extensions of their identity, not just product pages.
 
The goal wasn't just to build a webpage. It was to recreate that feeling — the weight of a luxury brand's digital presence — using only beginner-level CSS tools. Every decision, from typography to spacing to hover behavior, was made with one question in mind: *does this feel premium?*
 
Main focus areas:
- Aesthetic UI that mirrors high-end fashion brands
- Elegant typography with custom font integration
- Smooth, intentional user experience
- Fully responsive layout across all screen sizes
---
 
## ✨ Features
 
- **Responsive design** — adapts cleanly across Desktop, Tablet, and Mobile
- **Custom Didot font integration** — chosen specifically for its editorial, fashion-magazine feel
- **Hover effects** — subtle interactions that add life without JavaScript
- **Sticky mobile add-to-cart section** — keeps the purchase action always accessible on smaller screens
- **Custom favicon** — small detail, big difference in how polished the tab feels
- **Clean, minimal UI** — no clutter, no noise; every element earns its place on the page
---
 
## ⚠️ Real Challenges I Faced
 
**1. No Flexbox, No Grid, No JavaScript**
These weren't part of the curriculum yet. The entire layout had to be built using only positioning, margins, and paddings — which becomes genuinely complex once you try to make it responsive. What sounds simple in theory becomes a puzzle in practice.
 
**2. Main product image breaking on mobile**
The hero product image — the centerpiece of the whole page — kept collapsing or overflowing on smaller screens. Getting it to behave consistently across breakpoints took multiple rewrites of the responsive section.
 
**3. Overuse of `position: absolute`**
Early in the build, I relied too heavily on absolute positioning to place elements. This caused cascading layout issues as screen sizes changed. Walking it back and rethinking the structure mid-project was frustrating but necessary.
 
**4. Cross-browser inconsistencies**
Chrome was breaking layouts that looked perfectly fine in Brave and Safari. Debugging across browsers — without DevTools giving the same results everywhere — added unexpected time to the project.
 
**5. Making mobile feel *premium*, not just *functional***
Responsiveness that merely works isn't enough for a luxury concept. Making the mobile experience feel intentional and refined — not like a desktop layout that got compressed — required a lot of careful debugging and visual judgment calls.
 
---
 
## 📚 What I Learned
 
- **Responsive design is a mindset, not just media queries** — you have to plan for small screens from the very beginning, not bolt it on at the end
- **Typography carries more weight than most beginners expect** — the right font alone can shift a page from generic to editorial
- **Custom font integration** — how to correctly load and apply fonts that aren't in the standard browser stack
- **Layout planning matters** — starting to code without a clear structure in mind leads to exactly the kind of `position: absolute` chaos I ran into
- **Frontend debugging is a skill of its own** — reading what's breaking, forming a hypothesis, testing it, and iterating is something you only get better at by doing it under real pressure
